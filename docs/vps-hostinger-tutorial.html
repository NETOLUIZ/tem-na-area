Tutorial VPS Hostinger: Deploy Completo do Projeto Tem na Area

Guia de estudo com o processo que foi seguido para publicar frontend, backend e banco em uma VPS, com PM2, Nginx e Supabase.

1. Objetivo

Publicar um projeto full stack em uma VPS da Hostinger usando React + Vite no frontend, Node.js no backend, PostgreSQL no Supabase, PM2 para processo persistente e Nginx como servidor web e proxy reverso.

2. Arquitetura Final

Componentes

  • VPS Ubuntu na Hostinger
  • Nginx
  • PM2
  • Supabase Postgres
  • GitHub

Rotas finais

  • Frontend: http://IP_DO_SERVIDOR
  • API: http://IP_DO_SERVIDOR/api/v1

3. Problema Inicial

O primeiro erro foi ENOENT Could not read package.json. O motivo era simples: o comando npm estava sendo executado em uma pasta sem o projeto clonado.

cd /var/www/tem-na-area-api/tem-na-area
git pull origin main
npm install
npm start

4. Ajuste na Raiz do Repositório

Foram adicionados scripts na raiz para que npm start inicializasse a API correta dentro do workspace backend/node.

"scripts": {
  "build": "npm run build --workspace frontend",
  "start": "npm run start --workspace backend/node",
  "dev:api": "npm run dev --workspace backend/node",
  "check:api": "npm run check --workspace backend/node"
}

5. Banco de Dados no Supabase

Depois de corrigir a pasta, a API caiu ao tentar conectar em 127.0.0.1:5432. Isso mostrou que o backend ainda estava apontando para um banco local inexistente.

Foi criado o arquivo backend/node/.env com a URL do Supabase.

DATABASE_URL=postgresql://USUARIO:SENHA@aws-1-sa-east-1.pooler.supabase.com:5432/postgres
TEM_NA_AREA_DB_SSLMODE=require
TEM_NA_AREA_DB_SSL_REJECT_UNAUTHORIZED=false
NODE_TLS_REJECT_UNAUTHORIZED=0
TEM_NA_AREA_APP_ENV=production
TEM_NA_AREA_APP_URL=http://72.62.106.59
TEM_NA_AREA_APP_KEY=tem-na-area-chave-super-secreta-2026
TEM_NA_AREA_PORT=3001
CORS_ORIGIN=*
Nota: durante o processo houve erro de certificado SSL e por isso foi usado NODE_TLS_REJECT_UNAUTHORIZED=0 como ajuste provisório.

6. Publicando a API com PM2

cd /var/www/tem-na-area-api/tem-na-area
pm2 start npm --name temnaarea -- start
pm2 save
pm2 startup

Validacao:

pm2 status
curl http://127.0.0.1:3001/api/v1/health

7. Problemas com Porta 3001

Em alguns momentos apareceu EADDRINUSE, indicando que a porta 3001 ja estava ocupada. O diagnostico foi feito com:

lsof -i :3001
pm2 logs temnaarea --lines 50

Depois a API foi reiniciada corretamente no PM2.

8. Build do Frontend no VPS

cd /var/www/tem-na-area-api/tem-na-area/frontend
npm install
npm run build

Houve erro de dependencia opcional do Rollup no Linux:

npm install @rollup/rollup-linux-x64-gnu --save-dev
npm run build

Com isso foi gerada a pasta frontend/dist.

9. Configurando o Nginx

O arquivo /etc/nginx/sites-available/default foi alterado para servir o frontend e encaminhar a API.

server {
    listen 80 default_server;
    listen [::]:80 default_server;

    server_name _;

    root /var/www/tem-na-area-api/tem-na-area/frontend/dist;
    index index.html;

    location /api/ {
        proxy_pass http://127.0.0.1:3001/api/;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location / {
        try_files $uri $uri/ /index.html;
    }
}

Depois:

nginx -t
systemctl reload nginx

10. Ajuste no Frontend para Mesmo Host

O frontend foi alterado para usar /api/v1 em producao, mantendo http://127.0.0.1:3001/api/v1 no ambiente de desenvolvimento.

const defaultApiBaseUrl = import.meta.env.DEV ? "http://127.0.0.1:3001/api/v1" : "/api/v1";
const API_BASE_URL = (import.meta.env.VITE_API_BASE_URL || defaultApiBaseUrl).replace(/\/+$/, "");

13. Licoes Aprendidas

  • rodar npm na pasta certa e essencial
  • scripts de raiz ajudam muito em monorepo
  • 502 Bad Gateway geralmente e problema no upstream
  • sempre testar com curl antes de culpar o Nginx
  • copiar a string do Supabase inteira evita erro de credencial
  • servir frontend e API no mesmo host simplifica CORS e deploy

14. Proximos Passos
